# EviSI-Eval Shared Semantic Extraction Protocol

本协议同时约束源侧与目标侧的 Anchor、Event、Relation 抽取。两侧必须使用相同的类型定义、决策步骤、粒度规则和排除标准。抽取是文本分析，不是翻译正确性判断；不得根据另一侧文本猜测当前侧"应该出现"什么。

**信息隔离**：源侧只看源文，目标侧只看译文。目标侧不得猜测原文含义或判断译对译错。错误、额外或异常表达也要按当前文本原样抽取。

---

## 一、Anchor 协议

### 1.1 定义

Anchor 是文本中**具有独立核验价值的信息点**——即一个后续可以独立判断"是否准确传达"的信息单位。

Anchor 不是传统 NER 意义上的命名实体，也不是完整事件。Anchor 的本质是：一个**可以从命题中剥离出来、单独验证的事实性信息片段**。

### 1.2 类型体系

所有 anchor 必须归入以下 5 大类 13 子类：

| 大类 | 编码 | 子类 | 抽取条件 | 典型不可抽取 |
|---|---|---|---|---|
| **实体** | A-ENT | 人名/人物称谓/明确指代 | 文本中提及的具体个人或可识别群体 | 泛化人称（"someone"）、无 antecedent 的代词 |
| | A-ENT | 机构/组织/公司/政府/团队/项目 | 有名称或可识别的组织实体 | 普通概念（"the company" 无具体名） |
| | A-ENT | 地点/地理区域/场所 | 具体地名或可定位的地理范围 | 纯方位词（"here""there""左边"） |
| | A-ENT | 产品/政策/法规/活动/会议/文件名 | 有名称或可识别标识的项目/文件名 | 普通引用（"the report" 无具体名） |
| **量化** | A-QNT | 数字/金额/比例/百分比/排名 | 任何精确数字或含数字的范围 | 模糊量词（"一些""很多"） |
| | A-QNT | 度量单位+数量 | 出现在数字后的具体单位 | 纯单位词（"美元"无数字不抽） |
| | A-QNT | 范围/规模/序数/频率 | 含明确边界的范围表达 | 开放式描述（"很大的"不抽） |
| **时间** | A-TMP | 绝对时间（年月日/时刻） | 具体的日历时间或钟点 | 纯相对时间无参照（"后来"不抽） |
| | A-TMP | 相对时间/时段/频率 | 有明确参照的时间表达 | 纯模糊词（"很快""不久"） |
| **术语** | A-TERM | 专业术语/缩写/行业概念 | 领域特有的技术性词汇或缩写 | 日常同义词替换（"买"代替"收购"） |
| **限定** | A-SCOPE | 限定对象/群体/类别 | 带有明确分类限定的名词短语 | 无限定词的名词（"儿童"不抽，"三岁以下儿童"抽） |
| | A-SCOPE | 边界限定语（首次/至少/超过/约） | 修饰数字或范围的边界限定词 | 无量化对象的边界词（"至少"孤立出现不抽） |
| | A-SCOPE | 比较级/最高级限定 | 明确比较对象和范围的比较结构 | 无参照的比较（"更好"无比较对象不抽） |

### 1.3 判定决策树

对于每个候选文本片段 X，依次执行：

```
问题 1：X 是不是[实体/量化/时间/术语/限定]中的一类？
  → 否 → 不是 anchor
  → 是 → 继续

问题 2：X 是否可以脱离所在谓词结构、独立验证？
  例如："公司收入增长了 25 万美元"
  - "25 万美元" → 可以单独验证（金额是否正确）→ 是 anchor
  - "增长了" → 不能脱离"什么增长了"来验证 → 不是 anchor
  - "公司" → 可以单独验证（是不是这个公司）→ 是 anchor
  - "公司收入增长了 25 万美元" → 是完整事件，不是 anchor
  → 否 → 不是 anchor（可能是事件成分）
  → 是 → 继续

问题 3：X 是否提供了无法回避的核验义务？
  即：如果当前文本没表达 X，信息接收者是否会缺失关键事实信息？
  → 否 → 不是 anchor（可能是修饰成分）
  → 是 → 抽取为 anchor
```

### 1.4 Anchor 与 Event 的边界

这是最常见的混淆。遵循以下区分原则：

| 场景 | 归属 | 理由 |
|---|---|---|
| "25 万美元" 单独出现 | **Anchor** | 可独立验证的金额，不依赖谓词 |
| "收入增长了 25 万美元" | **Anchor: "25 万美元"** + **Event: "收入增长了 25 万美元"** | 金额是 anchor，完整事件是 event，两者在 evidence_span 上重叠，各自抽取 |
| "公司宣布了新政策" | **Anchor: "公司""新政策"** + **Event: "公司宣布了新政策"** | 实体 anchor + 完整事件，重叠抽取 |
| "增长了" 孤立出现 | **两者都不是** | 缺主语和对象，既不是 anchor（孤立动作词）也不是完整 event（不完整） |
| "很重要" | **两者都不是或事件的一部分** | 判断词本身不构成独立 anchor；如果"很重要"是完整判断事件的一部分，则归入 event |
| "欧洲市场" | **Anchor** | 实体地点，可独立验证 |
| "进入了欧洲市场" | **Anchor: "欧洲市场"** + **Event: "进入了欧洲市场"** | 实体 anchor + 完整事件 |

**关键原则：anchor 和 event 不是互斥的。** 同一个文本片段可以是 anchor（作为可核验信息点）同时又出现在 event 中（作为事件语义的组成部分）。两者应分别抽取。

### 1.5 抽取粒度规则

- **数字+单位**：必须整体抽取。"25 万美元"不能拆为"25"和"万美元"两个 anchor。
- **范围表达**：必须整体抽取。"30% 到 40%""至少 20 人"作为一个 anchor。
- **时间表达**：完整时间整体抽取。"2025 年 6 月""未来三年"。
- **限定对象**：带限定成分的对象整体抽取。"低收入家庭""三岁以下儿童"。
- **重复出现**：同一 anchor 在不同分析单元中分别抽取。同一单元内无信息增量的重复只抽一次。

### 1.6 不可抽取清单

| 失败模式 | 示例 | 原因 |
|---|---|---|
| 孤立动作/状态/判断词 | "增长""下降""宣布""认为""导致" | 没有独立核验价值，只有参与事件时才体现 |
| 逻辑连接词 | "因为""但是""如果""虽然""而且" | 不是信息锚点，属于 relation |
| 泛化代词 | "他""她""它""他们""这""那" | 除非从当前单元内可以稳定确定指向 |
| 口语填充词 | "嗯""啊""就是""然后""这个那个" | 无信息内容 |
| 完整事件或整句 | "公司收入增长了 25 万美元"整体 | 应拆分为 anchor + 事件 |
| 无具体指称的名词 | "公司""机构""项目""部门" 无具体名 | 除非上下文明确指向唯一可识别实体 |
| 纯情态/语气词 | "可能""必须""应该""大概" | 属于 event 的情态成分，不是独立 anchor |
| 无参照的相对词 | "很快""后来""之前""上面" | 除非有可确定的参照点 |

### 1.7 规范化规则

- 数字标准化："twenty-five million dollars" → "25 million dollars" 允许
- 单位统一化："25 million USD" → "2500 万美元" **不允许**（改变了货币类型）
- 日期换算：相对时间不强行换算为绝对日期
- **禁止**添加当前文本不包含的信息
- **禁止**把口语缩略/错误"纠正"后作为规范化含义

### 1.8 字段要求

- 引用所属分析单元 ID（源侧用 `source_unit_id`，目标侧用 `eval_unit_id`）。
- ID 按顺序连续编号（源侧 SA1/SA2/SA3，目标侧 TA1/TA2/TA3），不得重复。
- `anchor_type` 必须是 `A-ENT`、`A-QNT`、`A-TMP`、`A-TERM`、`A-SCOPE` 之一。
- `anchor_text` 是 anchor 的表面文本。
- `normalized_meaning` 轻度标准化，不得加入当前文本没有的信息。
- `evidence_span` **必须**是对应分析单元中**逐字**出现的连续片段。
  - ⚠️ **不得**清理填充词。当前文本 "the um project" → evidence_span 可以是 "the um project" 或 "um project"，**不能**写成 "the project"。
  - ⚠️ **不得**纠错。拼写错误 → 保留错误原文。
- 如果某个分析单元没有可抽取 anchor，不需要为该单元输出空记录。
- 源侧必须输出 `importance`（1/2/3），目标侧禁止输出 importance。

## 二、Event 协议

### 2.1 定义

Event 是文本中表达的**最小完整命题语义单位**。每个 event 对应一个可以被验证为"是否传达了"的命题。

### 2.2 类型体系

所有 event 必须归入以下类型之一。按优先级从高到低匹配，当一个命题同时符合多个类型时，取优先级最高的：

| 优先级 | 编码 | 类型 | 定义 | 结构 | 示例 |
|---|---|---|---|---|---|
| 1 | E-SPEECH | 言说事件 | 主体说了、传达了、询问、请求、承诺了某个内容 | speaker + speech-verb + (content) | "他说项目已经完成"、"Are you done?" |
| 2 | E-JUDG | 判断事件 | 主体表达判断、态度、观点、评价、建议 | judge + judgment + (topic) | "专家认为风险很大"、"This approach is clever" |
| 3 | E-MODAL | 情态事件 | 可能性、必要性、意愿是命题核心，改变现实性或义务性 | modality + proposition | "可能还会增长"、"必须立即行动" |
| 4 | E-CHANGE | 变化事件 | 某种属性或状态发生了变化 | subject + change-verb + (delta) | "收入增长了 15%"、"价格下降了" |
| 5 | E-ACT | 动作事件 | 主体执行某个动作，影响某个对象 | agent + action + (patient) | "公司宣布了新政策" |
| 6 | E-STATE | 状态事件 | 主体处于某种状态或具有某种属性 | subject + state/attribute | "市场不稳定"、"成本很高" |
| 7 | E-REL | 关系事件 | 两个实体之间存在身份、隶属、组成等实体关系 | entity1 + rel-predicate + entity2 | "A 是 B 的子公司" |

**优先级使用**：先判断是不是言说行为（提问、回答、建议、请求等言语行为），再判断是不是判断/评价，然后才是情态、变化、动作、状态、实体关系。不要因为句中存在普通动作词就忽略问句、言说或判断功能。

**问句必须保留言语行为**："Are you done?" → E-SPEECH（询问），不能抽成 E-ACT（"完成设计"的动作）。

**观点必须保留立场**："I think X" → E-JUDG（判断），canonical meaning 保留 "speaker believes X"，不能把 X 写成无条件事实。

### 2.3 边界确定规则

```
规则 1：默认边界 = 单句或单句中的独立子句
  "公司收入增长了 15%，但成本也上升了 10%。"
  → Event 1: "收入增长了 15%"
  → Event 2: "成本上升了 10%"

规则 2：主从复合句可以拆分为多个 event
  "他说公司将削减成本。"
  → Event 1: "他说……"（主语=他，动作=说，内容=公司将削减成本）
  → Event 2: "公司将削减成本"（主语=公司，动作=削减，对象=成本）
  ⚠️ 注意：不要过度拆分。如果从句只是修饰成分（如定语从句），可以不拆。

规则 3：并列结构应拆分
  "我们开发了新系统并培训了员工。"
  → Event 1: "开发了新系统"
  → Event 2: "培训了员工"

规则 4：残句/不完整句
  即使句子不完整，也要如实抽取当前可识别的事件语义。
  "因为所以……就是……" → 不抽取（无完整命题）
  "然后他就……走了" → Event: "他走了"（evidence_span="他就……走了"）

规则 5：同一个分析单元中多个事件
  如果同一单元中有多个独立命题，每个命题作为一个 event。
  不要为了保持"一个单元只有一个 event"而把不相关的信息合并。
```

### 2.4 内容要求

每个 event 应该包含该命题的核心语义要素：

- **言说事件**：说话者 + 说话内容（问句保留询问功能）
- **判断事件**：判断者 + 判断内容（评价保留立场）
- **情态事件**：情态范围 + 修饰的命题
- **变化事件**：主体 + 变化动词 ≈ 变化量
- **动作事件**：主体 + 动作 ≈ 对象/方向
- **状态事件**：主体 + 状态描述
- **关系事件**：关系双方 + 关系类型

**允许省略**的内容：时间修饰语、地点修饰语、程度副词（除非程度是核心语义）。

### 2.5 Completeness 原则

- 一个分析单元中的所有命题都应该被至少一个 event 覆盖。
- 如果一个分析单元包含 3 个独立命题，你应该输出 3 个 event，而不是 1 个或 2 个。
- **不要选择性抽取**——只抽"重要的"事件而忽略其他。
- 例外：纯口语填充序列（"然后就是……嗯……所以……"）不包含任何命题，不抽 event。

### 2.6 Event 与 Anchor 的关系

**关键规则：event 和 anchor 是正交的抽取任务。**

| 文本 | Anchor 抽取 | Event 抽取 |
|---|---|---|
| "公司收入增长了 25 万美元" | "公司"(A-ENT), "25 万美元"(A-QNT) | "公司收入增长了 25 万美元"(E-CHANGE) |
| "2023 年在上海举办了峰会" | "2023 年"(A-TMP), "上海"(A-ENT), "峰会"(A-ENT) | "在上海举办了峰会"(E-ACT) |
| "必须立即停止" | 无（"立即"无参照，"停止"孤立动作） | "必须立即停止"(E-MODAL) |

- Anchor 抽取的是**可核验的信息点**。
- Event 抽取的是**可判断传递性的命题**。
- 两者在文本上可以重叠，但抽取逻辑不同，互不依赖。

### 2.7 字段要求

- 引用所属分析单元 ID（源侧用 `source_unit_id`，目标侧用 `eval_unit_id`）。
- ID 按顺序连续编号（源侧 SE1/SE2/SE3，目标侧 TE1/TE2/TE3），不得重复。
- `event_type` 必须是七种类型之一。
- `event_text` 是事件语义的简洁描述（可以是原文片段，也可以是简明的语义描述）。
- `canonical_meaning` 是事件的规范化含义，可以用更清晰的方式表达，但不能加入当前文本没有的信息。必须保留否定、方向、情态和立场。
- `evidence_span` 必须是对应分析单元中逐字出现的连续片段（与被 anchor 重叠的情况可以共享跨度）。
- 本步骤不使用 anchor 抽取结果。
- 如果某个分析单元没有可抽取 event，不输出空记录。
- 源侧必须输出 `importance`（1/2/3），目标侧禁止输出 importance。

## 三、Relation 协议

### 3.1 定义

Relation 是事件之间、命题之间或信息片段之间的逻辑关系，包括因果、条件、转折、让步、目的、时序、比较、归因、解释、例外、递进等。

**默认没有 Relation。** Relation 是稀疏的意义结构，不是句子相邻图。没有充分证据时输出空数组。

### 3.2 类型列表

`relation_type` 仅可取以下值：

`cause_effect`、`condition_consequence`、`purpose`、`concession`、`contrast`、`temporal_sequence`、`temporal_overlap`、`conjunction`、`progression`、`similarity`、`difference`、`degree`、`elaboration`、`attribution`、`exemplification`、`exception`、`conclusion`。

### 3.3 Relation 成立流程

```text
是否存在至少两个不同的已抽取 Event？
  否 → 不抽

是否存在显式关系 cue（because/but/if/although/所以/因为/但是/虽然 等）？
  是 → 验证：
    1. cue 确实在文本中逐字存在
    2. cue 连接的是两个命题（不是只连接名词短语或仅作为话语填充）
    3. 关系方向与 cue 的语义一致
    全部通过 → 抽取为 explicit_cue
    任一不通过 → 不抽
  否 → 继续

是否存在唯一稳定的语义蕴含关系，且置信度 >= 0.85？
  检查方法（回译测试）：把两端 Event 和关系 paraphrase 成一个连贯的复句，复句是否
  (a) 自然通顺，且 (b) 与原两句的语义组合精确一致？
  是 → 抽取为 strong_semantic_entailment
  否 → 不抽
```

### 3.4 强制排除

以下情况自身**绝不构成** Relation：

1. A 在文本中先出现、B 后出现（不是 temporal_sequence）。
2. 问句后面出现回答（这是对话结构，不是 Relation）。
3. 一个说话者结束后另一个说话者开始。
4. 两句讨论同一主题/同一概念，但第二句不解释第一句的具体含义（不是 elaboration）。
5. B 只是 A 之后的另一条信息，没有逻辑连接（不是 conjunction）。
6. 两句内容不同但没有在同一比较维度上的对立（不是 contrast）。
7. 出现 `and` 但只是短语连接或话语填充，不是连接两个完整命题。
8. 出现 `so`/`then` 但只是口语组织词（"so basically..."、"and then..."），未表达实际因果或时序。
9. 出现 `you know`/`I mean`/`like` — 这些是话语组织词，不是关系 cue。
10. 为了让每个相邻单元都有边而抽取 temporal_sequence 或 elaboration。

### 3.5 字段要求

- 引用所属分析单元 ID（源侧用 `source_unit_ids`，目标侧用 `eval_unit_ids`）。
- ID 按顺序连续编号（源侧 SR1/SR2/SR3，目标侧 TR1/TR2/TR3），不得重复。
- `relation_type` 必须使用 §3.2 的 snake_case 类型名。
- `relation_basis`：`explicit_cue` 或 `strong_semantic_entailment`。
- `relation_cue`：explicit_cue 时，**逐字**复制原文中的连接词/短语，必须在所选单元文本中存在；strong_semantic_entailment 时，必须是**空字符串** `""`。
- `confidence`：0 到 1 之间。explicit_cue 通常为 0.90–0.95。strong_semantic_entailment **必须 ≥ 0.85**，低于此值不得输出。
- `relation_text` 是关系的简洁描述。
- `relation_meaning` 是关系的规范化含义，包含方向和类型。
- `evidence_spans` 中的每个 span 必须能在对应分析单元中找到逐字证据。explicit_cue 时数组至少包含 cue 本身 + 两端各至少一个 span。
- `related_*_event_ids`（源侧 `related_source_event_ids`，目标侧 `related_target_event_ids`）必须至少绑定两个不同 Event ID，必须来自本次输出中的 Event。无法稳定绑定时不抽取 Relation，不能编造 event_id。
- 源侧必须输出 `importance`（1/2/3），目标侧禁止输出 importance。

## 四、共同自检清单

输出前请逐项确认：

**无损/逐字**
1. 所有分析单元按顺序拼接是否等于输入文本？
2. 每个 `evidence_span` 是否在其对应单元中**逐字**存在？没有做清理、纠正或标准化？

**ID 连续性**
3. 所有 ID 是否各自从 1 开始无重复顺序编号？

**Anchor 质量**
4. 每个 anchor 是否通过了判定决策树的 3 个问题？（可分类→可独立验证→有核验义务）
5. 是否避免了常见错误？——完整事件当 anchor、孤立动作词当 anchor、无 antecedent 代词当 anchor、填充词当 anchor、无参照相对词当 anchor？
6. 数字+单位是否整体抽取？范围表达是否整体抽取？限定对象是否整体抽取？
7. 如果同一信息既有 anchor 又有 event，是否两者都抽取了？（重叠不是问题，不抽才是）

**Event 质量**
8. 每个 event 是否包含了一个完整的命题语义（符合对应 event 类型的结构要求）？
9. 类型选择是否按优先级顺序匹配（E-SPEECH → E-JUDG → E-MODAL → ...）？
10. 同一分析单元中的多个独立命题是否都被覆盖了（completeness 原则）？
11. `event_text` 和 `canonical_meaning` 对同一事件的描述是否语义一致？canonical_meaning 是否保留了否定、方向、情态和立场？

**Relation 质量**
12. 每个 Relation 是否连接了至少两个不同 Event 并有明确的方向和依据（explicit_cue 或 confidence ≥ 0.85）？
13. 是否把相邻出现、问答、话题延续、对话轮次或口语组织词误当成 Relation？
14. explicit_cue 时 relation_cue 是否在原文中逐字存在？strong_semantic_entailment 时 relation_cue 是否为空字符串？
15. 所有 `related_*_event_ids` 是否引自已输出的 Event？没有编造？

**信息隔离**
16. 是否没有输出 score 或 judgement 字段？
17. 源侧：每个项目是否都有合法的 `importance`（1/2/3）？目标侧：是否没有输出 importance？
18. 没有合格项目的类别是否诚实输出了空数组？
