# EviSI-Eval v0.6 完整技术方案

> 状态：已实现
> 协议版本：`evisi_eval_v0.6`
> 实现版本：`0.6.1`
> 核心方法：Source 事实义务 + Reference/SI 源文条件化投影 + 确定性计分

## 1. 目标

EviSI-Eval v0.6 用于评估同传系统的最终译文。系统首先从源语 transcript 建立可核验的 Anchor、Event 和 Relation；参考译文与同传译文不再自由建立独立语义卡，而是拿着已经冻结的 Source Items，在各自文本中逐项寻找、抽取和解释对应表达。

除 Fluency 和 SI Expression 外，核心内容维度都要求：

- 每项判断对应明确 Source Item；
- 目标证据逐字可定位；
- 错误译法也必须被抽取，不能直接当作 missing；
- Source 是事实权威；
- Reference 只是目标语辅助表达，不是标准答案字符串；
- 分数由 Python 根据结构化状态确定性计算。

## 2. 最终设计原则

### 2.1 Source 是唯一事实权威

Source Card 在任何 Reference/SI 投影前建立并冻结。Reference 不得修改、删除或新增 Source obligation。Reference 与 Source 冲突时，只能把 Reference 项标成 contradiction。

### 2.2 Reference 和 SI 都是源文条件化投影

Reference/SI Agent 接收 Source Card 和目标译文。对每个 Source Anchor/Event/Relation 恰好生成一条 projection，并记录目标证据与映射状态。

这不是盲抽取，也不是按参考译文字符串匹配。投影过程以 Source Item 为查询：

```text
Source Item
  -> 在对齐目标单元中寻找相关表达
  -> 错误表达仍抽取 target span
  -> 比较结构化语义组成
  -> 输出 equivalent/partial/contradiction/missing/uncertain
```

### 2.3 Reference 只辅助 SI

SI Agent 可以看到 Reference Projection，但必须遵守：

1. SI 与 Reference 不同不自动构成错误；
2. 首先判断 SI 是否与 Source 等价；
3. Reference 只有在其自身与 Source equivalent 时才提供强辅助；
4. Reference 为 partial/contradiction/missing/uncertain 时不能用来否定 SI；
5. 只有明确硬约束才能要求固定形式或精确组成。

### 2.4 不维护译法白名单或黑名单

协议禁止 `accepted_forms`、`rejected_forms`、`allowed_aliases` 和 `forbidden_aliases`。只记录少数明确的单个硬约束：

- 唯一官方目标语形式；
- 数值、币种、单位、日期和范围；
- Event 必须保持的谓词、角色、否定、情态或方向。

### 2.5 错误所有权

- Anchor：身份、数值、单位、时间、术语、范围；
- Event：核心谓词、必要角色、否定、情态、方向、立场；
- Relation：端点 Event 间的关系类型与方向；
- Fluency：目标语语法和自然度；
- SI Expression：同传特有的表达效率问题。

同一根因不能在依赖层重复扣分。

### 2.6 冻结卡独立，Context 只负责溯源

Source Card 和 Reference Projection Card 分别冻结、分别缓存。Python 在两张卡均通过校验后构建 `Evaluation Context Card`，记录 `source_card_hash`、`reference_card_hash` 和逐项 ID 映射。Context 不调用 LLM、不复制语义结论，也不把 Reference 提升为事实权威。SI Card 必须记录 `evaluation_context_hash`，确保其所有语义输入可追溯。

## 3. 完整流程

```text
样本级，一次构建后供所有系统共享：

Phase 1  SourceSegmentAgent
         source_text -> 约 1-3 句的无损 source_segments

Phase 2  SourceAnchorAgent
         source_segments -> source_anchors

Phase 3  SourceEventAgent
         source_segments + source_anchors -> source_events

Phase 4  SourceRelationAgent
         source_segments + source_events -> source_relations
         -> 冻结 Source Obligation Card

Phase 5  TargetAlignmentAgent(reference)
         source_segments + reference_translation -> target_units

Phase 6  ReferenceAnchorProjectionAgent
Phase 7  ReferenceEventProjectionAgent
Phase 8  ReferenceRelationProjectionAgent
         -> 冻结 Reference Projection Card

Phase 9  Python EvaluationContextBuilder
         source/reference hashes + item ID links -> 冻结 Evaluation Context Card

每个 sample × system：

Phase 10  TargetAlignmentAgent(SI)
Phase 11  SIAnchorProjectionAgent
Phase 12  SIEventProjectionAgent
Phase 13  SIRelationProjectionAgent
Phase 14  FluencyAgent
Phase 15  SIExpressionAgent
Phase 16  Python deterministic scoring
```

## 4. 输入格式

### 4.1 samples.jsonl

```json
{
  "sample_id": "sample_001",
  "source_text": "Company received 250,000 US dollars.",
  "reference_translation": "公司收到了25万美元。",
  "reference_type": "human_gold",
  "src_lang": "en",
  "tgt_lang": "zh",
  "domain": "finance",
  "hard_requirements": [
    {
      "source_span": "Ascend",
      "requirement_type": "exact_target_form",
      "required_target_form": "昇腾",
      "reason": "官方产品名称"
    }
  ]
}
```

`reference_type` 建议值：

- `human_gold`：人工核验标准译文；
- `professional_offline`：专业离线译文；
- `machine_offline`：机器离线译文，默认值。

v0.6 要求 `source_text` 与 `reference_translation` 非空。`hard_requirements` 可选；只有用户/数据集提供的要求才能使用 `verified_input` basis。

### 4.2 outputs.jsonl

```json
{
  "sample_id": "sample_001",
  "system_name": "system_a",
  "si_translation": "公司收到了25万元。"
}
```

真实 `system_name` 不发送给 LLM，Agent 只看到 `anonymous_system`。

## 5. Source Card

### 5.1 Source Segment

Source Segment 是上下文窗口，不是 Event。通常包含约 2 句，允许 1-3 句弹性。所有 segment 按顺序拼接必须逐字符等于 source_text。

```json
{"segment_id":"G1","source_segment":"verbatim source context"}
```

参考译文和 SI 不自行产生另一套 segment 编号。TargetAlignmentAgent 把目标译文无损切分为 target_units，并通过 `source_segment_ids` 链接冻结的 G IDs。支持一对多、多对一、遗漏、新增和不确定对齐。

### 5.2 Source Anchor

```json
{
  "segment_id": "G1",
  "source_anchor_id": "SA1",
  "anchor_type": "A-QNT",
  "anchor_text": "250,000 US dollars",
  "normalized_value": "250000 USD",
  "components": {"value":"250000","currency":"USD"},
  "evidence_span": "250,000 US dollars",
  "importance": 3
}
```

Anchor 五类：A-ENT、A-QNT、A-TMP、A-TERM、A-SCOPE。数字与单位整体抽取；技术方法如 round-robin load balancing 必须是 A-TERM，不是 A-ENT。

### 5.3 Source Event

```json
{
  "segment_id": "G1",
  "source_event_id": "SE1",
  "event_type": "E-ACT",
  "evidence_spans": ["Company received 250,000 US dollars"],
  "predicate_span": "received",
  "core_predicate": "receive",
  "arguments": [
    {"role":"agent","surface_span":"Company","source_anchor_ids":[]},
    {"role":"value","surface_span":"250,000 US dollars","source_anchor_ids":["SA1"]}
  ],
  "operators": {
    "negation": false,
    "modality": null,
    "direction": "gain",
    "polarity": "positive",
    "stance": null
  },
  "canonical_proposition": "company receives a monetary value",
  "importance": 3
}
```

Event 使用结构化缩句，不把整个句子复制为 event_text。逐字证据、谓词、论元、operators 与规范化命题同时保留。

### 5.4 Source Relation

Relation 默认不存在，必须至少连接两个不同 Event。相邻、问答、话题延续和文本顺序本身均不构成 Relation。

```json
{
  "source_relation_id":"SR1",
  "segment_ids":["G1"],
  "relation_type":"cause_effect",
  "relation_basis":"explicit_cue",
  "relation_cue":"because",
  "confidence":0.96,
  "relation_text":"SE1 causes SE2",
  "relation_meaning":"A is stated as the cause of B",
  "evidence_spans":["A","because","B"],
  "related_source_event_ids":["SE1","SE2"],
  "importance":3
}
```

## 6. Target Alignment

```json
{
  "target_unit_id":"T1",
  "source_segment_ids":["G1"],
  "target_text":"公司收到了25万元。",
  "alignment_status":"aligned",
  "reason":"目标内容对应 G1"
}
```

所有 target_text 按顺序拼接必须逐字符等于完整目标译文。允许同一 source segment 被多个 target unit 引用，因此不会因同传延迟或拆句而被强制一对一。

## 7. Anchor Projection

每个 Source Anchor 恰好一条 Anchor Projection。

```json
{
  "projection_id":"AP1",
  "source_anchor_id":"SA1",
  "target_unit_ids":["T1"],
  "target_spans":["25万元"],
  "target_meaning":"250000 CNY",
  "component_results":[
    {"component":"value","source_value":"250000","target_value":"250000","status":"preserved","target_span":"25万"},
    {"component":"currency","source_value":"USD","target_value":"CNY","status":"contradicted","target_span":"元"}
  ],
  "mapping_status":"contradiction",
  "hard_requirement":{
    "required":true,
    "requirement_type":"exact_value_unit",
    "required_target_form":null,
    "required_semantics":[],
    "basis":"intrinsic_exactness",
    "reason":"金额和币种必须精确保持"
  },
  "hard_requirement_satisfied":false,
  "confidence":0.99,
  "reason":"数值保持但币种发生冲突"
}
```

`25万美元 -> 25万元` 必须是 contradiction，不允许用 partial 缓和明确的币种冲突。

## 8. Event Projection 与错误去重

Event Projection 只比较：

- core_predicate；
- 必要论元角色；
- negation、modality、direction、polarity、stance。

Anchor 的值或单位错误不改变 Event verdict：

| Source | SI | Anchor | Event |
|---|---|---|---|
| 收获25万美金 | 收获25万元 | contradiction | equivalent |
| 收获25万美金 | 售货25万美金 | equivalent | contradiction |
| 收获25万美金 | 售货25万元 | contradiction | contradiction |

Event 的 hard requirement 不使用固定句子，只列必须保持的语义字段：

```json
{
  "required":true,
  "requirement_type":"required_event_semantics",
  "required_target_form":null,
  "required_semantics":["core_predicate","direction"],
  "basis":"intrinsic_exactness",
  "reason":"动作与方向必须保持"
}
```

## 9. Relation Projection 与依赖去重

Relation 先检查端点 Event：

- 任一端点 missing/contradiction/uncertain：`blocked_by_event + not_scored`；
- 两端 equivalent/partial：再比较 Relation。

这样 Event 缺失不会再次产生 Relation 扣分。

```json
{
  "projection_id":"RP1",
  "source_relation_id":"SR1",
  "target_unit_ids":[],
  "target_spans":[],
  "target_meaning":null,
  "dependency_status":"blocked_by_event",
  "mapping_status":"not_scored",
  "confidence":0.98,
  "reason":"SE1 已缺失，关系不能独立判断"
}
```

## 10. 硬约束规则

### 10.1 类型

- `exact_target_form`：唯一官方目标语形式；
- `exact_value_unit`：值、单位、币种、时间、范围；
- `required_event_semantics`：Event 必须保持的结构字段。

硬约束默认关闭。普通 Event 仍参与 fidelity 评价，但只有用户明确指定，或错误会改变行动、风险、法律/医疗/金融含义、资格、禁止/义务、否定或方向时，才启用 `required_event_semantics`。

### 10.2 Basis

- `verified_input`：数据输入明确提供；
- `intrinsic_exactness`：数值、单位、否定、方向等天然精确事实；
- `model_inference`：模型推断，结果必须 provisional_review_required。

### 10.3 示例：昇腾

如果输入提供：

```json
{"source_span":"Ascend","requirement_type":"exact_target_form","required_target_form":"昇腾"}
```

则 SI 写成“升腾”违反 exact_target_form。系统不会额外生成“可接受形式”或“不可接受形式”列表。

## 11. Prompt 体系

协议按任务模块化：

```text
v06_anchor_protocol.md
v06_event_protocol.md
v06_relation_protocol.md
v06_projection_protocol.md
```

角色 Prompt 只加载当前任务需要的模块。例如：

```text
SI Anchor = Anchor Protocol + Projection Protocol + SI Anchor role
Source Event = Event Protocol + Source Event role
SI Relation = Relation Protocol + Projection Protocol + SI Relation role
```

因此规则是同一文件，不依赖复制同步；同时不会把全部协议注入每个调用。组合后的 Prompt hash 记录到 manifest。

## 12. Validation

Python 只做可确定验证：

- ID 连续和引用存在；
- Source/Target 无损拼接；
- 每个 Source Item 恰好一条 projection；
- evidence/target spans 逐字存在；
- Event Anchor 引用合法；
- Relation 至少两个端点；
- implicit Relation confidence >=0.85；
- missing 不得引用目标证据；
- blocked Relation 必须 not_scored；
- 硬约束字段合法；
- resume 的输入、模型、Prompt、实现 hash 完全一致。

结构验证失败时 Runner 最多做两次 schema repair；仍失败则记录到 failures.jsonl，不生成语义 fallback。

## 13. 确定性计分

### 13.1 权重

```text
Anchor Fidelity    35
Event Fidelity     35
Relation Fidelity  10
Fluency            12
SI Expression       8
```

Anchor 与 Event 共占 70%，Relation 降为 10%。权重是 v0.6 初始参数，后续必须根据人工标注集校准。

### 13.2 Mapping 值

```text
equivalent     1.0
partial        0.5
contradiction  0.0
missing        0.0
uncertain      不进入分母
not_scored     不进入分母
```

每个内容维度：

```text
dimension_score = 100 * sum(importance * mapping_value) / sum(decided importance)
```

没有 Source Items 的维度为 not_applicable，不进入单样本总分权重。所有项目 uncertain 或 blocked 时维度为 no_decisions，最终分数为 null。

### 13.3 Delivery

```text
minor      -2
moderate   -6
major      -15
critical   -35
```

### 13.4 Score Status

- `final`：所有适用内容维度都有决定，且没有 model_inference 硬约束；
- `provisional_review_required`：存在 uncertain 或 model_inference 硬约束；
- `provisional_no_decisions`：某个适用内容维度没有可决定项目。

系统正式平均分只包含 `final`，provisional 单独聚合。

## 17. 客观性边界

结构化证据、错误所有权和确定性计分提高了可审计性，但 LLM 的语义等价判断仍不是天然客观真值。正式 Benchmark 发布前必须完成：

1. Source Anchor/Event/Relation 人工金标；
2. Reference Projection 人工核验；
3. SI 投影的人类标注者一致性；
4. 多模型/多次运行稳定性；
5. 权重和 partial 值校准；
6. model_inference 硬约束误判率审计。
