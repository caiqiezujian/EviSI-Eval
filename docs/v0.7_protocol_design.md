## 1. 设计哲学

### 1.1 核心原则

| 原则 | 含义 |
|:---|:---|
| **Source 唯一语义权威** | Source 原文是判断 SI 同传正确性的唯一标准。Reference 参考译文仅起辅助作用，SI ≠ Reference 绝非自动扣分 |
| **LLM 负责语义，代码负责编排** | LLM 承担所有语义判断（抽取、匹配、评估）；代码层负责流程编排、结构性校验、I/O 和确定性计算。代码永远不判断翻译对错 |
| **联合抽取 + 位置匹配** | Source 和 Reference 一同抽取、按数组索引对齐（第 i 条 source ↔ 第 i 条 reference ↔ 第 i 条 SI），不依赖 ID 交叉引用 |
| **Joint Card 一次性冻结** | 每个 Sample 的 Source+Reference 联合抽取只做一次，冻结后所有 SI 系统共享。冻结使用 SHA-256 内容哈希，保证不可篡改 |
| **确定性计分** | 所有评分规则在 `calculate_v07_scores()` 中硬编码，基于 match 值（equivalent/partial/contradiction/missing/uncertain）按公开算法计算，无需 LLM 二次评判 |
| **自包含 Prompt** | 每份 `.md` prompt 独立完整，包含全部规则、示例和输出格式。无需运行时协议注入、无需外部模板拼接 |
| **Flat JSON 输出** | 所有 LLM 输出的 artifact 均为扁平数组结构，无嵌套 `component_results`、operators grid、`hard_requirement` 等复杂 schema |

### 1.2 设计约束

```
代码层职责边界：
✅ 流程编排（14 阶段调度）
✅ 结构性校验（ID 连续性、数组长度、evidence 逐字出现）
✅ I/O（JSON/JSONL 读写、缓存）
✅ 确定性计算（分数计算、加权聚合）
❌ 判断翻译对错
❌ 修改 LLM 输出的语义内容
❌ 给 reference/si 字段赋值默认语义判断
```

## 2. 协议总览

### 2.1 14 阶段流水线

```
                          ┌─── Source 侧 (4 calls) ───┐
                          │  Phase 1: Segment          │
                          │  Phase 2: Anchor           │
                          │  Phase 3: Event            │
Source Text ──────────────│  Phase 4: Relation         │
                          └────────────────────────────┘
                          ┌─── Reference 侧 (4 calls) ─┐
                          │  Phase 5: Align            │
                          │  Phase 6: Anchor           │
Reference Text ───────────│  Phase 7: Event            │
                          │  Phase 8: Relation         │
                          └────────────────────────────┘
                                          │
                              Python zip → Joint Card (SHA-256 冻结)
                                          │
                          ┌─── Per-SI System (6 calls) ┐
                          │  Phase 9:  Align           │
                          │  Phase 10: Anchor Match    │
                          │  Phase 11: Event Match     │
                          │  Phase 12: Relation Match  │
                          │  Phase 13: Fluency         │
                          │  Phase 14: SI Expression   │
                          └────────────────────────────┘
                                          │
                              Python calculate_v07_scores()
                                          │
                              最终分数 + 诊断详情
```

### 2.2 调用规模

```
每个 Sample（共享）: 8 次 LLM 调用 → Joint Card 冻结
每个 SI System:      6 次 LLM 调用 → 匹配 + 评分

总计: 8 + 6N 次 LLM 调用（N = SI 系统数）

示例: 10 samples × 3 systems = 8×10 + 6×10×3 = 260 次 LLM 调用
```

### 2.3 关键数据流

```
source_text ──→ Source Agents ──→ source_segments/anchors/events/relations
                                         │
ref_text ────→ Reference Agents ─→ reference_align/anchors/events/relations
                                         │
                              Python _assemble_joint_card():
                                - 按 seg_id zip 合并
                                - 生成 flat_anchors/flat_events flat_relations
                                - SHA-256 哈希封存
                                         │
si_text ─────→ SI Agents ──→ matches + delivery issues
                                         │
                              calculate_v07_scores():
                                - 逐 Anchor/Event/Relation 计分
                                - 加权聚合 5 维度
                                - 输出 final_score + diagnostics
```

## 3. 输入/输出数据格式

### 3.1 Samples 输入 (`samples.jsonl`)

每行一个 Sample，对应一段需要评测的源文+参考译文：

```json
{
  "sample_id": "en2zh-01-tech_001",
  "source_text": "Round-robin load balancing scheme...",
  "reference_translation": "轮询负载均衡方案...",
  "src_lang": "en",
  "tgt_lang": "zh",
  "domain": "tech"
}
```

**兼容别名**：
- `transcript` → `source_text`
- `offline_translation` → `reference_translation`
- `vid` → `sample_id`

**硬约束**：
- `source_text` 非空
- `reference_translation` 非空
- `sample_id` 非空且全局唯一

每行一个 (sample_id, system_name) 组合：

**硬约束**：
- `si_translation` 非空
- `(sample_id, system_name)` 组合全局唯一
- `sample_id` 必须存在于 samples 中

## 4. Phase 1-4: Source 侧抽取

Source 侧是**整个评测链路的唯一语义权威**。这 4 个 Phase 从源文中抽取出评测骨架，后续所有匹配都基于此骨架。

### 4.1 Phase 1: Source Segmentation

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/source/v07_source_segment.md` |
| **Prompt 名称** | `v07_source_segment` |
| **输入** | `{sample_id, source_text}` |
| **输出** | `source_segments: [{seg_id, source_text}]` |
| **Validator** | `validate_source_segments()` |

#### 任务

将 `source_text` 全文按语义断句为 ~2 句/段。同传场景的口语特征（填充、重复、修正、残句）如实保留。

#### 硬约束（Validator 强制校验）

1. **无损拼接**：`"".join(seg.source_text for seg in segments) == source_text`（逐字符相等，包括空格、换行、标点、异常字符）
2. **连续 ID**：`seg_id` 必须为 `S1, S2, S3, ...`（严格从 1 连续编号）
3. **无空段**：每个 segment 的 `source_text` 不得为空
4. **至少一段**：`source_segments` 数组不得为空

#### 信息隔离

此阶段**仅看到 source_text**，看不到任何译文（Reference/SI）。

---

### 4.2 Phase 2: Source Anchor Extraction

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/source/v07_source_anchor.md` |
| **Prompt 名称** | `v07_source_anchor` |
| **输入** | `{sample_id, source_segments}` |
| **输出** | `source_anchors: [{anchor_id, seg_id, type, text, evidence, importance}]` |
| **Validator** | `validate_source_anchors()` |

#### 任务

从 source_segments 中抽取事实槽位（Anchor），回答"谁、什么专名、多少、何时、哪个术语、什么范围"。

#### 5 类 Anchor 类型

| 类型 | 定义 | 示例 |
|:---|:---|:---|
| `entity` | 唯一身份或可识别对象 | `Dr. Li`, `Spotify`, `ECMWF` |
| `term` | 术语、方法和领域概念 | `data assimilation`, `finite-volume discretization` |
| `quantity` | 数量、数值和度量 | `18.4 MWh`, `30%`, `three times per day` |
| `temporal` | 时间锚点 | `June 30, 2026`, `last quarter` |
| `scope` | 独立范围和适用边界 | `licensed operators only`, `at least 20 people` |

#### 提取门控（三问检验）

每个候选 Anchor 必须同时通过：
1. **Q1 类型归属**：能否归入以上 5 类之一？否 → 可能是 Event 成分，不抽
2. **Q2 独立核验**：脱离当前谓词后，是否仍可被识别、归一或比较？否 → 只是 Event 内部成分
3. **Q3 信息义务**：若同传遗漏或改变此信息，听众是否会失去一个可定位的事实？否 → 普通修饰，不抽

#### 字段语义

| 字段 | 含义 | 约束 |
|:---|:---|:---|
| `anchor_id` | `{seg_id}_A{n}` 格式，段内从 A1 连续编号 | `S1_A1`, `S1_A2`, `S2_A1` |
| `seg_id` | 所属 segment | 必须存在于 source_segments |
| `type` | entity/term/quantity/temporal/scope | 五选一 |
| `text` | Anchor 的干净可读表面形式，去除填充词 | 非空 |
| `evidence` | segment 文本中的逐字连续子串，保留原文包括拼写错误和填充 | 必须逐字出现在 source_text 中 |
| `importance` | 1=背景, 2=重要, 3=关键 | 必须为 1/2/3 |

#### 关键边界规则

- **entity**：不抽无唯一身份的普通名词（`company`, `system`）、无 antecedent 的代词（`he`/`it`/`this`）
- **term**：保留完整固定搭配，`finite-volume discretization` 不拆分为两个 Anchor；反复出现的概念每次出现分别建 Anchor
- **quantity**：`18.4 megawatt-hours` 整体不拆分；`rose by 15%` 只抽 `15%`
- **scope**：孤立的 `only`/`at least` 不抽，必须是独立适用边界
- **跨段重复**：不同 segment 中再次出现的同一概念分别建立独立 Anchor
- **填充处理**：`um`/`uh` 不单独建 Anchor；被填充打断的术语合并为一个 term；修正以最终表达为准

#### 硬约束（Validator 强制校验）

1. `source_anchors` 为数组
2. `anchor_id` 格式正确且段内连续编号
3. `type` 在允许集合内
4. `text` 非空
5. `evidence` 逐字出现在对应 segment 的 `source_text` 中
6. `importance` ∈ {1, 2, 3}

### 4.3 Phase 3: Source Event Extraction

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/source/v07_source_event.md` |
| **Prompt 名称** | `v07_source_event` |
| **输入** | `{sample_id, source_segments}` |
| **输出** | `source_events: [{event_id, seg_id, type, summary, evidence, importance}]` |
| **Validator** | `validate_source_events()` |

#### 任务

从 source_segments 中抽取结构化事件。Event 描述"发生了什么"——谓语 + 语义角色（谓词、主体、对象、工具、来源、目标、受益者等）。

#### 4 类 Event 类型

| 类型 | 定义 | 示例 |
|:---|:---|:---|
| `action` | 带明确施事或因果影响的行为 | `company announces new product` |
| `state` | 属性、状态、关系或能力 | `product uses neural networks` |
| `speech` | 口头/书面表达（引用、陈述、提问） | `Mark mentioned looking at different metrics` |
| `judgment` | 评估、比较、意见（模态、倾向） | `It's fine to be honest when...` |

#### 字段语义

| 字段 | 含义 | 约束 |
|:---|:---|:---|
| `event_id` | `{seg_id}_E{n}` 格式，段内从 E1 连续编号 | `S1_E1`, `S2_E1` |
| `seg_id` | 所属 segment | 必须存在于 source_segments |
| `type` | action/state/speech/judgment | 四选一 |
| `summary` | 事件摘要（可包含语义角色） | 非空 |
| `evidence` | segment 文本中的逐字连续子串 | 必须逐字出现 |
| `importance` | 1=背景, 2=重要, 3=关键 | 必须为 1/2/3 |

#### Anchor vs Event 区分

```
"Dr. Li mentioned that data assimilation uses Bayesian methods."
  → Anchor (entity): Dr. Li
  → Anchor (term): data assimilation, Bayesian methods
  → Event (speech): Dr. Li mentions data assimilation uses Bayesian methods
```

#### 硬约束

1. `event_id` 格式正确且段内连续
2. `type` ∈ {action, state, speech, judgment}
3. `summary` 非空
4. `evidence` 逐字出现在对应 segment 中
5. `importance` ∈ {1, 2, 3}

### 4.4 Phase 4: Source Relation Extraction

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/source/v07_source_relation.md` |
| **Prompt 名称** | `v07_source_relation` |
| **输入** | `{sample_id, source_segments, source_events}` |
| **输出** | `source_relations: [{relation_id, type, summary, source_event_ids, evidence, importance}]` |
| **Validator** | `validate_source_relations()` |

#### 任务

从 source_segments 和 source_events 中抽取事件之间的关系。

#### 16 类 Relation 类型

```
cause_effect        因果     "A causes B"
condition_consequence 条件-结果  "if A then B"
purpose             目的     "A to achieve B"
concession          让步     "Despite A, B"
contrast            对比     "A vs B"
temporal_sequence   时序先后  "A then B"
temporal_overlap    时序重叠  "A while B"
conjunction         并列     "A and B"
progression         递进     "Not only A, but also B"
similarity          相似     "A like B"
difference          差异     "A unlike B"
degree              程度     "A to the extent that B"
elaboration         阐述     "A, namely B"
attribution         归因     "According to A, B"
exemplification     例证     "A, for example B"
conclusion          结论     "A, therefore B"
```

#### 字段语义

| 字段 | 含义 | 约束 |
|:---|:---|:---|
| `relation_id` | `R{n}` 格式，全局连续编号 | `R1, R2, R3` |
| `type` | 16 类之一 | 必须在 RELATION_TYPES 集合内 |
| `summary` | 关系描述 | 非空 |
| `source_event_ids` | 关联的事件 ID 列表 | 至少 2 个，无重复，全部存在于 source_events |
| `evidence` | 源文中的逐字证据 | 非空（此处 evidence 可为跨段文本） |
| `importance` | 1-3 | 必须为 1/2/3 |

#### 硬约束

1. `relation_id` 全局连续（R1, R2, R3, ...）
2. `type` 在 16 类集合内
3. `source_event_ids` 至少 2 个，无重复，全部指向已知 event
4. `summary` 非空
5. `evidence` 非空
6. `importance` ∈ {1, 2, 3}

## 5. Phase 5-8: Reference 侧抽取

Reference 侧为 Source 的每个 item 提供目标语对应表达。Reference **不是标准答案**——它只是辅助 SI 评测时在目标语中定位内容的一种参考。

Reference 侧的核心约束：**每个 Reference item 必须与 Source 对应 item 按位置一一对应。** 数量和顺序必须完全一致。

### 5.1 Phase 5: Reference Alignment

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/reference/v07_reference_align.md` |
| **Prompt 名称** | `v07_reference_align` |
| **输入** | `{sample_id, source_segments, reference_translation}` |
| **输出** | `reference_segments: [{seg_id, reference_text}]` |
| **Validator** | `validate_reference_alignment()` |

#### 任务

将 `reference_translation` 全文切分为与 `source_segments` 数量完全相同的段，按语义内容对齐。

#### 硬约束

1. **数量相等**：`len(reference_segments) == len(source_segments)`
2. **ID 一致**：`reference_segments[i].seg_id == source_segments[i].seg_id`
3. **无损拼接**：`"".join(reference_segments[i].reference_text) == reference_translation`（逐字符相等）
4. **无空段**：每个 segment 的 `reference_text` 不得为空

---

### 5.2 Phase 6: Reference Anchor

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/reference/v07_reference_anchor.md` |
| **Prompt 名称** | `v07_reference_anchor` |
| **输入** | `{sample_id, source_segments, source_anchors, reference_segments, reference_translation}` |
| **输出** | `reference_anchors: [{anchor_id, seg_id, text, evidence}]` |
| **Validator** | `validate_reference_anchors()` |

#### 任务

对每一条 source_anchor，在对应 reference segment 中找到目标语对应表达。

#### 关键设计

- **数量和顺序与 source_anchors 完全一致**——第 i 条 reference anchor 对应第 i 条 source anchor
- 不输出 `type` 和 `importance`（Source 侧已定义，代码层在 Joint Card 组装时从 source 侧继承）
- Reference 中没有对应表达时，`text` 和 `evidence` 填空字符串 `""`
- 找的是**语义对应**，Reference 可能用不同于 Source 字面形式的表达

#### 字段语义

| 字段 | 含义 |
|:---|:---|
| `anchor_id` | 必须与对应 source anchor 完全相同 |
| `seg_id` | 必须与对应 source anchor 完全相同 |
| `text` | Reference 中对应表达的干净文本（无则 `""`） |
| `evidence` | Reference segment 中的逐字连续子串（无则 `""`） |

#### 硬约束

1. 数量 = source_anchors 数量
2. 位置一致：第 i 条 anchor_id 与 source_anchors[i].anchor_id 相同
3. seg_id 与对应 source anchor 相同
4. evidence 逐字存在于对应 reference segment 中

---

### 5.3 Phase 7: Reference Event

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/reference/v07_reference_event.md` |
| **Prompt 名称** | `v07_reference_event` |
| **输入** | `{sample_id, source_segments, source_events, reference_segments, reference_translation}` |
| **输出** | `reference_events: [{event_id, seg_id, type, summary, evidence, importance}]` |
| **Validator** | `validate_reference_events()` |

#### 任务

对每一条 source_event，在对应 reference segment 中找到目标语对应表达。

#### 关键设计

- 数量和顺序与 source_events 完全一致——第 i 条 reference event 对应第 i 条 source event
- 保留 source event 的 `type` 和 `importance`（Reference 可能使用不同的 summary 但 type/importance 不应改变——代码从 source 侧继承）

#### 硬约束

1. 数量 = source_events 数量
2. 位置一致：第 i 条 event_id 与 source_events[i].event_id 相同
3. seg_id 与对应 source event 相同

## 7. Phase 9-14: SI 侧匹配与评分

Joint Card 冻结后，每个 SI 系统执行 6 次 LLM 调用：4 次语义匹配 + 2 次表达评估。

### 7.1 Phase 9: SI Alignment

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/si/v07_si_align.md` |
| **Prompt 名称** | `v07_si_align` |
| **输入** | `{sample_id, source_segments, si_translation, system_name="anonymous_system"}` |
| **输出** | `si_segments: [{seg_id, si_text}]` |
| **Validator** | `validate_si_alignment()` |

#### 硬约束

1. `len(si_segments) == len(source_segments)` — 数量完全相等
2. 所有 seg_id 与 source_segments 一致
3. `"".join(si_text) == si_translation` — 无损拼接
4. 无空段

#### 同传特殊处理

- 同传有更多省略和压缩——segment 边界仍按 source 结构切分
- SI 对某段完全没翻译时，如实保留短段或连接词，不补内容
- 口语特征（重复、修正、不完整句）如实保留

### 7.2 Phase 10: SI Anchor Match

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/si/v07_si_anchor_match.md` |
| **Prompt 名称** | `v07_si_anchor_match` |
| **输入** | `{sample_id, source_segments, joint_anchors, si_segments, si_translation}` |
| **输出** | `anchor_matches: [{anchor_id, match, si_text, si_evidence, brief}]` |
| **Validator** | `validate_si_anchor_matches()` |

#### 5 个 match 值

| match 值 | 分数 | 判定标准 |
|:---|:---:|:---|
| `equivalent` | 1.0 | SI 等价覆盖 Anchor：与 Reference 相同或等价表达，或不同于 Reference 但与 Source 语义等价 |
| `partial` | 0.5 | SI 覆盖核心但有关键缺失：实体可识别但不精确、术语不完整、数值对但缺单位 |
| `contradiction` | 0.0 | SI 与 Source 明确冲突：数值/单位错误、译成不同概念、实体错误、范围翻转 |
| `missing` | 0.0 | SI 中完全找不到该 Anchor |
| `uncertain` | — | 存在证据但无法稳定判断。**从分母排除，最终触发 provisional** |

#### 核心判断原则

**Source 是唯一语义权威，SI ≠ Reference 绝非自动扣分。**

```
Source: "data assimilation"
Reference: "数据同化"
SI: "数据融合" → 若该译法在目标语中也通用 → equivalent（不是错误）

Source: "finite-volume discretization"
Reference: "有限体积离散"
SI: "有限体积离散化" → equivalent（同义术语）
```

#### 字段语义

| 字段 | 含义 |
|:---|:---|
| `anchor_id` | 必须与对应 joint anchor 相同 |
| `match` | equivalent / partial / contradiction / missing / uncertain |
| `si_text` | SI 中的对应干净文本（无则 `""`） |
| `si_evidence` | SI segment 中的逐字连续子串（无则 `""`） |
| `brief` | 判断理由简述（≤50 字） |

#### 按类型匹配指引

| 类型 | 匹配关注点 | partial 示例 | contradiction 示例 |
|:---|:---|:---|:---|
| entity | 身份可识别性 | 称谓模糊但可推断 | 明显是不同的人/机构 |
| term | 技术概念准确传达 | 术语被简化 | 翻译成不同概念 |
| quantity | 数值等价+单位正确 | 数值对但缺单位 | 数值/单位/币种错误 |
| temporal | 时间点/持续期准确 | 精度丢失 | 时间明显错误 |
| scope | 限定词保留 | 边界模糊化 | 边界翻转（only→all） |

### 7.3 Phase 11: SI Event Match

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/si/v07_si_event_match.md` |
| **Prompt 名称** | `v07_si_event_match` |
| **输入** | `{sample_id, source_events, joint_events, si_segments, si_translation}` |
| **输出** | `event_matches: [{event_id, match, si_summary, si_evidence, brief}]` |
| **Validator** | `validate_si_event_matches()` |

#### 匹配标准（与 Anchor 相同 5 值体系）

- **equivalent**: SI 等价表达了事件的核心命题：谓词相同或等价、关键语义角色完整
- **partial**: 核心命题在但有关键角色丢失或弱化（如主语省略但可推断）
- **contradiction**: 命题与 Source 明确冲突（动作方向错误、否定/情态翻转、对象张冠李戴）
- **missing**: SI 中完全找不到该事件的任何痕迹
- **uncertain**: SI 片段不完整导致无法判断

#### 关键原则

- 同传有压缩是正常的——省略次要角色但仍传达了核心命题 → equivalent 或 partial
- Event 匹配判断的是**命题是否被传达**，不是逐词对等
- Anchor 值错误（如数值译错）不应影响 Event 的判分——各维度独立

### 7.4 Phase 12: SI Relation Match

| 属性 | 值 |
|:---|:---|
| **Prompt 文件** | `prompts/si/v07_si_relation_match.md` |
| **Prompt 名称** | `v07_si_relation_match` |
| **输入** | `{sample_id, source_relations, joint_relations, si_event_matches, si_translation}` |
| **输出** | `relation_matches: [{relation_id, match, brief}]` |
| **Validator** | `validate_si_relation_matches()` |

#### 特殊 match 值：`not_scored`

`not_scored` 是 relation 维度独有的状态。当 relation 关联的端点 event 全部为 missing 或 contradiction 时，该 relation 标记为 `not_scored`，**不参与计分**（从分母排除）。

**设计理由**：如果端点事件都已丢失或被矛盾替换，再去评判关系是否保留没有意义——这不是双倍惩罚，而是逻辑上无法评估。

#### Validator 强制校验

```
关系声明 not_scored 
  → 验证其所有 source_event_ids 对应的 event_match 确实都是 missing 或 contradiction
  → 否则报错 "not_scored but not all endpoints are missing/contradiction"
```

## 8. Runner 调度器：调用→验证→修复→兜底

### 8.1 执行流程

```
Runner.run(prompt_name, payload, validator, fallback=None)

步骤 1: LLM 调用
  client.generate_json(load_prompt(prompt_name), payload)
  → artifact (from response.data)

步骤 2: 结构验证
  issues = validator(artifact)
  if no issues → return StageResult

步骤 3: 修复循环 (最多 MAX_REPAIR_ATTEMPTS=2 次)
  for attempt in 1..2:
    if no issues → break
    repair = client.generate_json(load_prompt("schema_repair"), {
      "stage_name": prompt_name,
      "stage_input": payload,
      "validation_issues": issues,
      "repair_attempt": attempt,
      "json_to_repair": artifact,
    })
    artifact = repair.data
    issues = validator(artifact)

步骤 4: 兜底
  if issues and fallback:
    artifact = fallback()
    issues = validator(artifact)

步骤 5: 失败
  if issues → raise ValueError(f"{prompt_name} failed validation: {issues}")
```

### 8.2 Schema Repair

`prompts/schema_repair.md` 是一个通用修复 prompt。它接收：
- `stage_name`: 原始阶段名
- `stage_input`: 原始输入
- `validation_issues`: 校验发现的具体错误
- `json_to_repair`: 需要修复的 JSON

修复只针对**结构问题**（ID 格式、数组长度、evidence 逐字性、字段缺失等），从不重新做语义判断。

### 8.3 修复次数统计

每个 `StageResult` 记录 `initial_issues` 和 traces，包括：
- 首次调用的 trace（`prompt_name` task）
- 每次修复的 trace（`repair_{prompt_name}` task）
- 兜底 trace（`fallback_{prompt_name}` task）

## 10. 确定性评分模型

### 10.1 函数入口

```python
calculate_v07_scores(
    joint_anchors, joint_events, joint_relations,
    anchor_matches, event_matches, relation_matches,
    fluency_issues, expression_issues,
) → {dimension_scores, dimension_weights, score_diagnostics, final_score, score_status}
```

### 10.2 Fidelity 维度（Anchor / Event / Relation）

核心算法 (`_match_dimension`)：

```python
for each item in joint_items:
    match_value = anchor_matches[id_key].match
    
    if match_value == "uncertain":        # 排除
        uncertain += 1; continue
    if match_value in skip_statuses:      # relation 的 not_scored
        blocked += 1; continue
    
    score_value = STATUS_VALUES[match_value]  # equivalent=1.0, partial=0.5, contradiction/missing=0.0
    contribution = importance × score_value
    earned_weight += contribution
    decided_weight += importance

if decided_weight == 0:
    if blocked == len(joint_items) and uncertain == 0:
        → "blocked_by_dependency" (relation 全被阻塞, 不适用)
    else:
        → "no_decisions" (所有项 uncertain, final_score = null)

score = 100 × earned_weight / decided_weight
```

**STATUS_VALUES:**
| match 值 | 分数值 |
|:---|:---:|
| `equivalent` | 1.0 |
| `partial` | 0.5 |
| `contradiction` | 0.0 |
| `missing` | 0.0 |
| `uncertain` | 排除 |
| `not_scored` (relation only) | 排除 |

**importance 加权**：每个 item 的 importance (1-3) 作为权重，关键信息 (importance=3) 比背景信息 (importance=1) 对分数影响更大。

**示例计算**：
```
Anchors: SA1(imp=2, equiv=1.0) + SA2(imp=2, partial=0.5) + SA3(imp=3, missing=0.0)
  earned = 2×1.0 + 2×0.5 + 3×0.0 = 3.0
  decided_weight = 2 + 2 + 3 = 7.0
  anchor_fidelity = 3.0 / 7.0 × 100 = 42.86
```

### 10.3 Relation 依赖阻断

```
if relation endpoints E1 and E2 are both missing:
  → relation is "not_scored"
  → excluded from denominator
  → no double penalty for events that don't exist
```

Validated by `validate_si_relation_matches()`: 声明 `not_scored` 的 relation 必须所有端点都是 missing 或 contradiction。

### 10.4 Delivery 维度（Fluency / SI Expression）

```python
def _delivery_score(issues):
    deduction = sum(SEVERITY_DEDUCTIONS[issue.severity] for issue in issues)
    return max(0.0, 100.0 - deduction)
```

**SEVERITY_DEDUCTIONS:**
| severity | 扣分 |
|:---|:---:|
| minor | 2.0 |
| moderate | 6.0 |
| major | 15.0 |
| critical | 35.0 |

每个 issue 的 `target_span` 重复只扣一次（validator 强制校验 `target_span` 去重）。

起点 100 分，逐项扣减。

### 10.5 维度权重

```python
DIMENSION_WEIGHTS_V07 = {
    "anchor_fidelity": 35,
    "event_fidelity":   35,
    "relation_fidelity": 10,
    "fluency":          12,
    "si_expression":     8,
}
# 总和 = 100
```

**最终分数**：
```python
final_score = round(
    sum(dimension_scores[dim] × weight for dim in active_dimensions)
    / sum(weight for dim in active_dimensions),
    2
)
```

只有 `applicable=True` 的维度参与聚合。例如：如果一个 sample 中所有 relation 都是 `not_scored`，relation_fidelity 的 `applicable=False`，不会拖累总分。

### 10.6 Score Status

| Status | 条件 |
|:---|:---|
| `final` | 所有 fidelity 维度无可疑项（no uncertain items） |
| `provisional_review_required` | 至少一个 fidelity item 状态为 uncertain（需要人工复审） |
| `provisional_no_decisions` | 所有 fidelity items 都是 uncertain → `final_score = null` |

### 11.2 缓存机制

```python
def _run_cached_stage(runner, prompt_name, payload, validator, cache_path, resume):
    input_hash = artifact_hash(payload)
    
    if resume and cache_path.exists():
        cached = read_json(cache_path)
        # 验证 prompt_name 和 input_hash 匹配
        if cached["prompt_name"] != prompt_name or cached["input_hash"] != input_hash:
            raise ValueError("stage cache does not match current input")
        # 重新跑 validator 确保完整性
        issues = validator(cached["artifact"])
        if issues:
            raise ValueError("stage cache failed validation")
        return StageResult(artifact=cached["artifact"], ...)
    
    # 正常执行
    result = runner.run(prompt_name, payload, validator)
    if cache_path:
        write_json(cache_path, {
            "prompt_name": prompt_name,
            "input_hash": input_hash,
            "artifact": result.artifact,
            "traces": result.traces,
            "initial_issues": result.initial_issues,
        })
    return result
```

### 11.3 Resume 机制

```
run_v07_pipeline(..., resume=True):

1. 读取 run_manifest_v07.json
2. 对比当前配置（prompt hashes, model, weights）
   → 任何变化 → ValueError: "Cannot resume — configuration changed"
3. 读取已缓存的 joint_cards（校验 hash）
4. 读取已完成的 si_cards 和 results
5. 只执行未完成的 stages
```

**Resume 兼容性检查** (`_assert_resume_compatible`):
```
manifest 中记录：
  - protocol_version
  - implementation_version
  - implementation_hash (6 个核心 .py 文件的 SHA-256)
  - samples_sha256 / outputs_sha256
  - 所有 v07 prompt 的 SHA-256
  - provider / model
  - dimension_weights

任何一项变化 → 拒绝 resume
```

### 11.4 Failure 追踪

`failures.jsonl` 记录所有失败的 stage：
```json
{"stage": "joint", "sample_id": "en2zh-01-tech_001", "error": "v07_source_event failed validation: ..."}
{"stage": "si", "sample_id": "en2zh-01-tech_001", "system_name": "sys_a", "error": "..."}
```

- Resume 时从 failures 中过滤已成功的项
- 最终输出时去重（每个 (stage, sample_id, system_name) 只保留一条）

## 12. 信息隔离设计

### 12.1 隔离矩阵

| Agent | 看到 Source | 看到 Reference | 看到 SI | 输出 |
|:---|:---:|:---:|:---:|:---|
| Source Segment | ✅ | ❌ | ❌ | source_segments |
| Source Anchor | ✅ | ❌ | ❌ | source_anchors |
| Source Event | ✅ | ❌ | ❌ | source_events |
| Source Relation | ✅ (segments + events) | ❌ | ❌ | source_relations |
| Reference Align | ✅ (segments) | ✅ (full text) | ❌ | reference_segments |
| Reference Anchor | ✅ (segments + anchors) | ✅ (segments + full) | ❌ | reference_anchors |
| Reference Event | ✅ (segments + events) | ✅ (segments + full) | ❌ | reference_events |
| Reference Relation | ✅ (relations) | ✅ (events + full) | ❌ | reference_relations |
| SI Align | ✅ (segments) | ❌ | ✅ (full text) | si_segments |
| SI Anchor Match | ✅ (segments + joint) | ✅ (joint 内含) | ✅ (segments + full) | anchor_matches |
| SI Event Match | ✅ (events + joint) | ✅ (joint 内含) | ✅ (segments + full) | event_matches |
| SI Relation Match | ✅ (relations + joint) | ✅ (joint 内含) | ✅ (full) | relation_matches |
| Fluency | ❌ | ❌ | ✅ (full) | fluency_issues |
| SI Expression | ✅ (full) | ❌ | ✅ (full) | expression_issues |

### 12.2 关键隔离原则

1. **Source agents 永远看不到任何译文**——保证 Source 抽取的独立性
2. **Fluency agent 只看 SI**——评估语言流畅度而非翻译质量
3. **SI Expression agent 看 Source + SI**——评估同传表达效率
4. **SI matching agents 只看 Joint Card + SI**——Reference 只是辅助参考

### 12.3 匿名化

所有 prompt payload 中 `system_name` 统一设为 `"anonymous_system"`。真实系统名由代码层在输出时恢复。

## 附录 C: 关键设计决策记录

1. **Source 唯一权威** vs 双权威 (Source+Reference 同等)：选择单一权威——Source。Reference 的语言表达可能也有偏差，以其为标准会在 SI 正确但不同于 Reference 时误判。
2. **位置匹配** vs ID 交叉引用：选择位置匹配。避免 ID 漂移导致的复杂交叉校验，数组索引天然保证一一对应。
3. **Flat JSON** vs 嵌套结构：选择 flat。减少 LLM 输出复杂度，降低 schema 错误率，方便 LLM 直接遍历。
4. **Stage 级缓存** vs Joint Card 整体缓存：选择 stage 级。任何一个阶段失败不丢失前面阶段的成果。Event 重抽不用重抽 Anchor。
5. **stdlib urllib** vs SDK：选择 stdlib。零依赖、可在受限网络环境中运行、升级不受 SDK 变更影响。
6. **单次匹配** vs 双模型复核（Primary+Reviewer+Adjudicator）：选择单次。v0.7 的匹配任务比 v0.5/v0.6 的评分任务更原子化（逐项比较而非整体判断），单次 LLM 判断可靠度已足够。
7. **schema_repair** 只做结构修复不做语义重判：修复 prompt 只针对 ID 格式、数组长度、evidence 逐字性等结构问题，绝不要求 LLM 重新做语义判断。
