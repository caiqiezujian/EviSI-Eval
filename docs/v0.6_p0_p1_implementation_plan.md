# v0.6.1 客观评测链路实施方案

> 更新日期：2026-06-30  
> 状态：已实现并通过自动化测试  
> 协议版本：`evisi_eval_v0.6`  
> 实现版本：`0.6.1`

## 1. 本次重构结论

本版本不建立 `Joint Authority Card`。Source 是唯一语义权威，Reference 只是冻结的辅助投影，两者必须保持独立，不能在数据命名或结构上暗示 Reference 拥有事实裁决权。

本版本新增由 Python 构建的 `Evaluation Context Card`：

```text
Source Obligation Card ────── source_card_hash ──┐
                                                 ├─ Evaluation Context Card
Reference Projection Card ─ reference_card_hash ┘       │
                                                        ├─ evaluation_context_hash
                                                        ▼
                                              SI Projection Card
```

Context Card 不复制或修改 Source/Reference 内容，不调用 LLM，只负责：

1. 绑定两张已冻结卡的 hash；
2. 显式记录 Source item ID 与 Reference projection ID 的对应；
3. 为 SI Card 提供完整输入溯源；
4. 保持 Source/Reference 独立缓存和断点恢复能力。

## 2. 被否决的原方案

### 2.1 不合并 Source 和 Reference 权威

原方案认为两张卡靠数组索引对应。实际实现已经按 `source_anchor_id`、`source_event_id` 和 `source_relation_id` 校验全集与唯一性，因此不存在只靠数组位置绑定的问题。

Source/Reference 继续分别冻结和缓存。Context 只建立显式链接，不改变权威边界。

### 2.2 不把决定性 Anchor 冲突改成 0.25

Anchor component 的任一明确冲突仍控制整体状态：

```text
任一 contradicted -> contradiction -> 0.0
无冲突但存在 omitted -> partial/missing -> 0.5/0.0
全部 preserved -> equivalent -> 1.0
存在无法稳定判断的证据 -> uncertain -> 不进入分母并触发复核
```

例如 `250,000 USD -> 25万元`：value preserved、currency contradicted，因此 Anchor 必须为 contradiction，不能用 partial 缓和。

### 2.3 不引入未经标注校准的 Event 常数

本版本不使用 `0.70/0.60/0.40` 等经验分值。谓词、论元和 operator 子状态进入审计诊断，但正式分值继续由整体 mapping status 确定。只有获得人工标注集和一致性统计后，才允许调整 partial 或子项权重。

### 2.4 不合并聚焦 Agent 调用

Source 的 Segment、Anchor、Event、Relation 继续分四次调用；Reference 的 Alignment、Anchor、Event、Relation 继续分四次调用。Context 由代码生成，不增加 LLM 调用。

## 3. 完整执行流程

### 3.1 样本级共享阶段

```text
Phase 1  SourceSegmentAgent
Phase 2  SourceAnchorAgent
Phase 3  SourceEventAgent
Phase 4  SourceRelationAgent
         -> Source Obligation Card + source_card_hash

Phase 5  TargetAlignmentAgent(reference)
Phase 6  ReferenceAnchorProjectionAgent
Phase 7  ReferenceEventProjectionAgent
Phase 8  ReferenceRelationProjectionAgent
         -> Reference Projection Card + reference_card_hash

Phase 9  Python Context Builder（零 LLM 调用）
         -> Evaluation Context Card + evaluation_context_hash
```

### 3.2 每个样本 × 同传系统

```text
Phase 10  TargetAlignmentAgent(SI)
Phase 11  SIAnchorProjectionAgent
Phase 12  SIEventProjectionAgent
Phase 13  SIRelationProjectionAgent
Phase 14  FluencyAgent
Phase 15  SIExpressionAgent
Phase 16  Python deterministic scoring
```

SI Projection Card 必须同时记录：

```json
{
  "source_card_hash": "...",
  "reference_card_hash": "...",
  "evaluation_context_hash": "...",
  "metadata": {"si_card_hash": "..."}
}
```

任一 hash 与当前冻结输入不一致，恢复流程立即拒绝旧卡。

## 4. 客观校验加强

### 4.1 Anchor

- Source components 必须是非空原子值，不能嵌套对象或数组；
- Projection 必须覆盖每个 Source component 恰好一次；
- `source_value` 必须与冻结 Source component 相同，LLM 不得重新归一化；
- preserved/contradicted 必须提供目标值和逐字 target span；
- omitted 不得携带目标值或伪造证据；
- 任一 component contradicted 必须控制整体 contradiction。

### 4.2 Event

- Source argument 必须有 role、逐字 surface span 和合法 Anchor 链接；
- argument 不允许重复；
- Source/Target operators 固定使用 `negation/modality/direction/polarity/stance`；
- Target Event 非 missing 时必须有核心谓词、规范命题和可定位证据；
- Anchor 的值错误不改变 Event 的谓词、角色和 operator 判断。

### 4.3 Relation

Relation 继续遵守端点依赖：任一端点 Event 不可用时，Relation 为 `blocked_by_event + not_scored`，避免同一根因重复扣分。

## 5. 计分与诊断

正式 mapping 值保持：

```text
equivalent     1.0
partial        0.5
contradiction  0.0
missing        0.0
uncertain      不进入分母
not_scored     不进入分母
```

每个内容维度的 `score_diagnostics.item_decisions` 新增：

- Source item ID；
- Projection ID；
- importance；
- mapping status；
- score value；
- weighted contribution；
- Anchor component statuses 或 Event substatuses。

这使报告能够解释“哪一项贡献了多少分”，但不会在没有人工校准前改变正式分数。

## 6. 输入数据

现有数据无需改写。v0.6 归一化器同时支持：

| 标准字段 | 兼容字段 |
|---|---|
| `source_text` | `transcript` |
| `reference_translation` | `offline_translation` |
| `sample_id` | `vid` |

系统输出使用 `sample_id + system_name + si_translation`。`system_asr` 会被忽略，不进入评测。

离线校验现有数据：

```bash
python -m evisi_eval check-v06-input \
  --samples data/user_samples.jsonl \
  --outputs data/user_two_files_si_only_outputs.jsonl
```

该命令不读取 API Key，也不调用大模型。

只运行第一个样本和第一个系统输出：

```bash
python -m evisi_eval run-v06 \
  --samples data/user_samples.jsonl \
  --outputs data/user_two_files_si_only_outputs.jsonl \
  --provider deepseek \
  --output-dir results \
  --run-name v061_smoke \
  --limit-samples 1 \
  --limit-outputs 1
```

## 7. 输出结构

```text
results/<run_name>/
├── source/source_cards_v06.jsonl
├── reference/reference_projection_cards.jsonl
├── context/evaluation_context_cards.jsonl
├── target/si_projection_cards.jsonl
├── score/final_results_v06.jsonl
├── failures.jsonl
├── metrics_v06.json
└── run_manifest_v06.json
```

Source、Reference 和 Context 分别缓存。Reference 或 SI 失败不会迫使 Source 阶段重新调用。

2. 多模型、多次运行一致性；

4. partial 值和五维权重校准；
5. model-inference hard requirement 的误判率；
6. Reference 类型对辅助可信度的影响。
