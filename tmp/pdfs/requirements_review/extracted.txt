EviSI-Eval Agent 需求与实施方案 v0.1
1. 项目定位
EviSI-Eval Agent 是一个面向同声传译最终译文的自动化质量评估 Agent。EviSI-Eval 是 Evidence-driven
Simultaneous Interpretation Evaluation 的缩写，即“证据驱动的同声传译评估”。
本系统的目标是评价同传系统最终输出的译文质量，而不是评价同传系统的实时中间过程。输入是一段源语转
录文本和一个同传系统的最终译文，系统通过大模型完成切分对齐、关键 anchor 抽取、事件抽取、逻辑关系抽
取、表达质量评判、内容忠实度评判和最终打分。
当前版本只评估最终文本，不评估音频质量、ASR 质量、真实延迟、partial revision、字幕稳定性、增量输出
过程或语音播报效果。
系统的核心目标是回答一个问题：听众通过这段同传译文，是否获得了与源文一致、足够完整、清楚自然、符
合同传表达特点的信息。
2. 总体原则
本系统是大模型驱动的评估 Agent，不是硬编码规则评分器。
所有核心语义判断由大模型完成，包括原文与译文切分、anchor 抽取、event 抽取、relation 抽取、源译对应
判断、表达问题判断和维度评分。
代码层只负责流程编排、Prompt 调用、JSONL 读写、字段合并、失败重试、格式校验、无损切分校验、日志
记录和报告生成。代码不负责判断翻译是否正确，也不负责用字符串规则判定 anchor、event 或 relation 是否
匹配。
为了让大模型评估尽量客观，系统必须采用结构化流程。不能让大模型直接读取原文和译文后自由给一个总
分。必须先生成可审查的中间结果，再基于这些中间结果进行评分。
评估过程中可以使用真实译文或离线译文作为参考，但参考译文只能辅助理解源文大意、术语表达和句子边
界，不能作为判断同传译文正确与否的证据。最终评估依据必须是源文和同传译文。

5.1 Anchor Fidelity，30 分
评估源文中的关键 anchor 是否在同传译文中被准确传达。
这里的 anchor 不等于传统 NER 实体，而是“关键可核验信息锚点”。它包括人名、机构、地点、时间、数
量、金额、比例、术语、项目名称、政策名称、对象范围、限定对象等。
例如：“25万美元”中的 25万 或 25万美元 是 anchor；“上升”不是 anchor，而是 event 中的变化方向。
anchor 只负责可独立核验的信息锚点，不负责动作、变化、态度或逻辑关系。
5.2 Event Fidelity，25 分
评估源文中的核心事件语义是否被同传译文准确保留。
event 关注“谁做了什么”“什么发生了变化”“谁对谁产生影响”“某人表达了什么判断或态度”“某状态
是否成立”等语义结构。
event 包括主体、动作、状态、变化方向、判断、态度、否定、情态、主客体关系、施事受事关系和范围边界
等。
5.3 Relation Fidelity，20 分
评估源文中的逻辑关系是否被同传译文准确保留。
relation 关注事件之间或命题之间的关系，包括因果、条件、转折、让步、目的、时序、比较、归因、解释、
例外、递进等。
relation 依赖 event 抽取结果。只有先知道两边各自有哪些事件，才能判断事件之间的关系是否保留。
5.4 Fluency，15 分
评估完整同传译文本身是否清楚、自然、可理解。
fluency 只看完整 si_translation ，不按切分单元逐句评估，也不看源文。因为流利度是目标语整体听感和可
理解性问题。单句局部不完整不一定影响整体理解，单句看似通顺也不代表整段自然清楚。
fluency 评估内容包括语法是否混乱、表达是否自然、句子是否可理解、是否存在源语残留、指代是否清楚、目
标语搭配是否异常、整体是否影响听众理解。
5.5 SI Expression，10 分
评估完整同传译文是否符合同传表达要求。
SI expression 关注同传译文作为实时口译结果是否简洁、顺畅、有效，是否存在无意义重复、过度填充、拖
沓、反复改述、无必要解释、明显不必要添加等问题。
SI expression 可以看完整 source_text 和完整 si_translation ，但它不负责判断核心内容是否忠实。内容
忠实度由 anchor、event 和 relation 三个维度判断。SI expression 主要判断表达方式和同传输出形态。

6. 总体流程
完整流程如下：
1. 句子/句组级一对一切分对齐
2. 原文 anchor 抽取
3. 译文 anchor 抽取
4. 原文 event 抽取
5. 译文 event 抽取
6. 原文 relation 抽取
7. 译文 relation 抽取
8. 整体 fluency 评判
9. 整体 SI expression 评判
10. anchor 内容忠实度评判
11. event 内容忠实度评判
12. relation 内容忠实度评判
13. 内容忠实度全文复核
14. 五维分数计算
15. 总分计算与总结
其中第 1 步是建立局部比较范围。第 2 到第 7 步是结构化语义抽取。第 8 和第 9 步是译文表达质量评判。第 10
到第 12 步是内容忠实度评判。第 13 步是全文复核。第 14 和第 15 步是评分与报告生成。
内容忠实度采用“切分后的对齐单元为主，全文复核为辅”的原则。也就是说，anchor、event、relation 的基
础判断在 aligned_unit 内完成，但最终还要做全文级复核，检查跨单元补偿、术语一致性、延迟表达、重复
错误和局部判断冲突。

8. Step 1：句子/句组级一对一切分对齐
8.1 目标
将 source_text 和 si_translation 切分并对齐为一组一对一评估单元。每个单元包含一个源文句子或相邻
源文句组，以及一个译文句子或相邻译文句组。
第一步不做句内细切分，不抽 anchor，不抽 event，不判断译文对错，不打分。
8.2 关键原则
默认按句子切分。句子内部的定语从句、倒装结构、插入语、后置修饰、长宾语、状语、补语等不拆开。
必要时合并相邻句子为句组。如果原文 A、B 两个相邻句子在译文中表现为 B′、A′，则输出一个单元：
source_unit = A+B ， target_unit = B′+A′ 。不要强行输出 A 对 B′、B 对 A′。
翻错仍然对齐。源文说“增长 15%”，译文说“下降 50%”，仍然对齐为同一单元，后续步骤再判断错误。
切分必须无损。所有 source_unit 按输出顺序拼接后，必须等于输入的 source_text 。所有 target_unit
按输出顺序拼接后，必须等于输入的 si_translation 。
8.3 输出字段
{
"aligned_units":[
{
"unit_id":"U1",
"source_unit":"源文句子或相邻源文句组，可以为空",
"target_unit":"译文句子或相邻译文句组，可以为空",
"alignment_status":"aligned | source_omitted | target_addition | uncertain",
"reason":"简要说明切分对齐原因"
}
]
}

文句子或相邻源文句组 source_unit，以及一个译文句子或相邻译文句组 target_unit。
你只负责切分和对齐，不抽取 anchor，不抽取事件，不判断译文对错，不打分。
输入字段包括：
sample_id
source_text
si_translation

reference_translation

tgt_lang
reference_translation 只能辅助理解源文大意、术语表达和可能的句子边界。不得把

2. 不要把一句话内部的定语从句、倒装结构、插入语、后置修饰、长宾语、状语、补语等拆成独立评估单

3. 如果严格按句子一对一会导致明显错误，可以将相邻源文句子合并为一个 source_unit，或将相邻译文
句子合并为一个 target_unit。
4. 如果源文 A、B 两个相邻句子在译文中表现为 B′、A′，应输出一个合并单元 source_unit = A+B，
target_unit = B′+A′。不要强行输出 A 对 B′、B 对 A′。
5. 如果译文片段明显是在表达某个源文句子，即使内容翻错，也应标为 aligned。内容错误由后续评估步
骤判断。
6. 如果某个源文句子或源文句组在译文中没有任何实质对应表达，则输出 source_omitted，并令
target_unit 为空字符串。
7. 如果译文中存在没有源文依据的独立句子或独立片段，则输出 target_addition，并令 source_unit
为空字符串。
8. 如果无依据添加嵌入在某个译文句子内部，不要在第一步强行切开；保留在对应 target_unit 中，后
续评估步骤再判断。
9. 切分不得省略、改写、纠错、清理或补全任何原文和译文内容。
10. 所有 source_unit 按输出顺序拼接后，必须等于输入的 source_text。
11. 所有 target_unit 按输出顺序拼接后，必须等于输入的 si_translation。
12. 如果原文句子和译文句子可以稳定对应，就不要合并。不要把整段原文和整段译文直接合成一个大单

9. Step 2 和 Step 3：Anchor 抽取
9.1 设计方式
Anchor 抽取使用同一个 Prompt 模板，调用两次。
第一次调用时 side = source ，只读取 source_unit ，输出 source_anchors 。
第二次调用时 side = target ，只读取 target_unit ，输出 target_anchors 。
源文 anchor 抽取和译文 anchor 抽取必须独立。抽源文 anchor 时不能看 target_unit 。抽译文 anchor 时不
能看 source_unit 。抽取阶段不判断对错，不比较原文和译文。
9.2 Anchor 定义
Anchor 是关键可核验信息锚点，不等于传统实体。Anchor 包括但不限于：
人名、机构、国家、城市、地点、时间、日期、年份、数量、金额、比例、百分比、度量单位、产品名、项目
名、政策名、文件名、专业术语、限定对象、明确指称的群体或对象。
Anchor 不负责抽动作、变化方向、因果、条件、态度、判断、否定、情态和逻辑关系。这些内容由 event 或
relation 阶段处理。
但 Prompt 中不需要写“这个交给 event 抽取”。Anchor Prompt 只需要正面说明自己抽什么，不需要引用
event 的职责。

10. Step 4 和 Step 5：Event 抽取
10.1 设计方式
Event 抽取使用同一个 Prompt 模板，调用两次。
第一次调用时 side = source ，只读取 source_unit ，输出 source_events 。
第二次调用时 side = target ，只读取 target_unit ，输出 target_events 。
Event 抽取必须独立于 anchor 抽取。即使输入 JSON 中已经存在 anchors，event Prompt 也不应使用 anchor
结果。Event Prompt 只看当前侧文本单元本身。
10.2 Event 定义
Event 是文本中表达的最小完整语义事件或状态判断。
Event 包括动作、状态、变化、判断、态度、言说行为、关系成立、影响发生等。
Event 需要尽量表达完整，不要只抽孤立动词。例如不要只抽“增长”，而应抽“收入增长了 15%”；不要只
抽“宣布”，而应抽“公司宣布新的投资计划”。

20. 关键设计约束
20.1 切分对齐阶段可以同时看原文和译文
因为切分对齐的任务是建立局部比较范围。它不是语义判断，也不是打分。
20.2 抽取阶段必须单侧独立
source anchor 抽取只看 source_unit。
target anchor 抽取只看 target_unit。
source event 抽取只看 source_unit。
target event 抽取只看 target_unit。
source relation 抽取只看 source_unit 和 source_events。
target relation 抽取只看 target_unit 和 target_events。
26
===== PAGE 27 =====
这样可以减少模型根据另一侧文本脑补内容的风险。
20.3 判断阶段才允许源译同时出现
anchor judgement、event judgement、relation judgement 阶段可以同时看源文结构和译文结构，因为这些
阶段的任务就是判断源译是否对应、是否等价、是否缺失、是否错误。
20.4 Fluency 是整体译文维度
fluency 不按 unit 打分，只看完整 si_translation 。它只评目标语整体清楚度、自然度和可理解性。
20.5 SI Expression 是整体同传表达维度
SI expression 看完整 source_text 和完整 si_translation ，判断译文作为同传输出是否简洁、顺畅、有
效。它不重复评判 anchor、event、relation 忠实度。
20.6 内容忠实度以切分单元为主，全文复核为辅
anchor、event、relation 的基础判断在 aligned_unit 内完成。全文复核只用于检查延迟表达、术语一致性、
跨句逻辑、重复错误和局部冲突。
20.7 最终评分不能新增错误
最终评分只能基于前面已经生成的结构化证据。评分阶段不能绕过前面流程重新自由判错。

21.2 每一步输入控制
虽然 JSONL 会滚动保留所有字段，但每次调用模型时最好只传当前 Prompt 需要的字段。
例如 source anchor 抽取时，只传 sample_id 、 aligned_units.unit_id 和 aligned_units.source_unit 。
不要把 target_unit 、 target_anchors 或后续字段传进去。
这样可以保证 Prompt 责任边界更清楚，也能减少模型信息泄漏。
21.3 格式校验
代码可以做以下格式校验：
1. JSON 是否可解析。
2. 必填字段是否存在。
3. unit_id 是否重复。
4. anchor_id、event_id、relation_id 是否重复。
5. evidence_span 是否出现在对应文本中。
6. 切分后 source_unit 拼接是否等于 source_text 。
7. 切分后 target_unit 拼接是否等于 si_translation 。
8. 分数是否在 0 到 100 之间。
9. 维度权重是否为固定值。
这些校验不是语义规则，不判断翻译对错，只保证输出结构可用。
21.4 失败处理
如果某一步输出 JSON 不可解析，重试该步骤。
如果切分无损校验失败，重试切分步骤。
如果 evidence_span 不存在于对应文本，重试对应抽取步骤。
如果连续重试仍失败，将该样本写入 bad cases 文件，不生成最终分数。
21.5 断点续跑
每一步都写独立 JSONL。已经完成的样本不重复调用模型。后续流程从最近可用的中间文件继续。
21.6 多系统评估
多个系统的评估应独立进行。每个系统的 si_translation 单独进入完整流程。

22. 当前版本不做的内容
当前版本不评估真实同传延迟。
当前版本不评估 partial 输出稳定性。
当前版本不评估字幕闪烁。
当前版本不评估音频质量。
当前版本不评估系统 ASR。
当前版本不使用时间戳。
当前版本不把参考译文作为判错依据。
当前版本不设置 anchor 或 event 的 1、2、3 重要程度。
当前版本不做传统实体类别分类。
当前版本不使用代码规则判断语义正确性。
23. 当前版本的核心结论
本方案的核心流程是：
先将原文和同传译文无损切分并对齐为句子/句组级一对一评估单元。
然后分别在原文侧和译文侧独立抽取 anchor、event 和 relation。
接着整体评判译文 fluency 和 SI expression。
之后在切分单元内完成 anchor、event、relation 的内容忠实度评判，并通过全文复核修正局部判断可能存在
的遗漏或冲突。
最后基于结构化结果给出五个维度分数，并按照 anchor 30、event 25、relation 20、fluency 15、SI
expression 10 的权重计算最终分数。
整个系统的客观性来自两个方面：第一，评估不是直接自由打分，而是经过切分、抽取、判断、复核、评分的
链式流程；第二，最终分数必须受前面结构化证据约束，不能在评分阶段重新任意新增错误。
